# Adoption Review comment template (canonical)

The canonical structure for the synthesis comment a maintainer posts when
adopting a contributor PR — running multi-model review on someone else's
PR, optionally pushing maintainer fixups, then merging.

Two forms cover all four merge paths the `pr-review` pack's `mol-adopt-pr`
distinguishes:

- **Form 1** (Path A): contributor's branch landed as-is, no maintainer commits.
- **Form 2** (Paths B / C / D): maintainer fixups present — pushed to the
  original PR (B), to a new maintainer-owned PR (C, when maintainer-edits
  is disabled), or to a follow-up PR (D, when the original was already merged).

This template is the reviewer-side counterpart to this pack's `mol-pr-review`
author-side outgoing-PR self-review report. Both produce structured findings
against the same 11-category scorecard so contributors get consistent
feedback shape regardless of direction.

The opener is friendly by structural requirement — adoption fixups can read
as direct critique without explicit framing, so the template enforces the
"thanks for getting this so far, we hardened it so others can get value out
of it" tone.

## Form 1 — No maintainer changes (Path A)

```markdown
## Adoption Review

Thanks for the contribution, @{contributor}! {fix-summary-1-2-sentences}.

### Review Summary
{decision-and-summary-2-3-sentences}.

### Review Findings
| # | Category | Severity | Confidence | Source | Summary |
|---|----------|----------|------------|--------|---------|
{findings-rows-or-literal-"No issues found."-line}

{N} {model-list}-driven review pass(es) performed.

---
_Adopted via `/adopt-pr` workflow. Contributor commits preserved._
```

## Form 2 — Maintainer changes present (Paths B, C, D)

```markdown
## Adoption Review

Thanks for the contribution, @{contributor}! {fix-summary-2-3-sentences}.

This PR was adopted with maintainer fixes pushed directly to the contributor's branch.

### Initial Review
{initial-review-paragraph}

### Maintainer Changes
{maintainer-changes-paragraph}

### Final Review Status
{final-review-paragraph-with-tip-sha-and-model-list}

### Review Iterations
{N} review pass(es) performed.

---
_Adopted via `/adopt-pr` workflow. Contributor commits preserved._
```

## Inputs (gathered before rendering)

| Field | Source | Notes |
|---|---|---|
| `contributor` | `gh pr view <N> --json author --jq .author.login` | |
| `pr-title` | `gh pr view <N> --json title --jq .title` | |
| `pr-body` | `gh pr view <N> --json body --jq .body` | |
| `initial-synthesis` | First review pass `<run_dir>/outputs/synthesis.md` | |
| `initial-scorecard` | First review pass `<run_dir>/outputs/scorecard.md` | |
| `final-synthesis` | Final review pass `<run_dir>/outputs/synthesis.md` | Same as initial if iteration_count=1 |
| `final-scorecard` | Final review pass `<run_dir>/outputs/scorecard.md` | |
| `maintainer-fixup-log` | `git log --format=%s ${ORIGINAL_HEAD}..HEAD` | One subject per line; empty for Path A |
| `maintainer-fixup-diff` | `git diff ${ORIGINAL_HEAD}..HEAD` | Empty for Path A |
| `final-tip-sha` | `git rev-parse --short HEAD` | Repo's `core.abbrev` (typically 7) |
| `model-list` | Bead notes `model_list` (set by review step) | E.g. `"Claude and Codex"` or `"Claude, Codex, and Gemini"` |
| `iteration-count` | Bead notes `iteration_count` (set by review step) | Default 1 |

## Rendering rules

Each `{model-rendered}` field below is generated by an LLM call with raw
inputs. The orchestration is mechanical (gather + validate + post); the
prose is the model's job. Per the Zero Framework Cognition principle, no
hardcoded sentence-construction logic outside the model.

- **`{fix-summary}`** — 1-3 sentences. Pattern: "This PR fixes a real
  {failure-mode} in the {subsystem}: {one-sentence-mechanism}." For Form 2,
  add a second sentence crediting the original direction and framing the
  fixups as hardening: "The original change targeted the right {area}, and
  the maintainer follow-up work hardened {what} so {value-statement}." For
  Form 1, drop the second sentence.

- **`{decision-and-summary}`** (Form 1 only) — 2-3 sentences. State the
  multi-model verdict, name which models agreed, summarize the bug-fix
  validity. Distinguish merge-blocking findings from follow-up-grade
  findings.

- **`{initial-review-paragraph}`** (Form 2 only) — 4-6 sentences. Walk
  initial-synthesis + initial-scorecard. State what initial review confirmed
  (the contributor's own report) and what additional findings came up
  (categorized by severity + area). Close with a one-sentence impact
  statement on why those findings counted as merge blockers.

- **`{maintainer-changes-paragraph}`** (Form 2 only) — 3-5 sentences. Walk
  maintainer-fixup-log + maintainer-fixup-diff. State what the fixups
  hardened in plain English (group related commits when possible). Then
  state what regression coverage was added. If lint/test cleanup was needed
  for green CI, mention it as a final clause.

- **`{final-review-paragraph}`** (Form 2 only) — 2-3 sentences. State the
  multi-model verdict on the final tip SHA (`final-tip-sha`), naming the
  models from `model-list`. End with the literal phrase
  "All clear — no remaining blocker or major issues found." iff the
  final-scorecard verdict is `approve`.

## Validation gates (enforced before posting — fail-stop)

A rendered comment is REJECTED if any of these fail. On rejection, write the
rejected text to `<run_dir>/outputs/adoption-review-rejected.md`, print
`[adopt-pr] Comment validation failed — see <path>`, and STOP. Do not post
a partial / malformed comment, do not auto-retry. The maintainer reads the
rejected file, edits, and posts manually.

### Both forms

1. Starts with `## Adoption Review` followed by a blank line and the verbatim
   opener `Thanks for the contribution, @` (the friendly tone is enforced
   structurally).
2. Ends with the literal footer line below (markdown-italic, with the
   slash-command name rendered as inline code — backticks are part of the
   literal):

   ```
   _Adopted via `/adopt-pr` workflow. Contributor commits preserved._
   ```

3. The `{fix-summary}` is 1-3 sentences (counted by terminal `.` `!` `?`
   outside of code/links).

### Form 1 (Path A) additional gates

4. Contains `### Review Summary` and `### Review Findings` headers in order.
5. Contains the iteration-count line ending in `review pass(es) performed.`
   with a positive integer matching root bead notes `iteration_count`.

### Form 2 (Path B/C/D) additional gates

4. Contains exactly one adoption-notice sentence — which one depends on the
   merge path. Validate the path-specific literal, not a single fixed string:

   - **Path B:** `This PR was adopted with maintainer fixes pushed directly to the contributor's branch.`
   - **Path C:** `This adopts the contribution from #<original-pr-number> on a maintainer-owned branch (the original PR did not have maintainer-edits enabled).`
     (with `<original-pr-number>` substituted to a real integer)
   - **Path D:** `This is a follow-up to #<original-pr-number> (already merged), bundling the maintainer fixups into a separate PR.`
     (with `<original-pr-number>` substituted to a real integer)

5. Contains four section headers in order: `### Initial Review`,
   `### Maintainer Changes`, `### Final Review Status`,
   `### Review Iterations`.
6. The `{final-review-paragraph}` contains a SHA prefix matching
   `git rev-parse --short HEAD` exactly.
7. The Review Iterations line ends in `review pass(es) performed.` with a
   positive integer matching root bead notes `iteration_count`.

## Path-specific adjustments

- **Path A** — Form 1. No adoption notice. Maintainer-fixup inputs are empty.
- **Path B** — Form 2. Posts on the original PR. Adoption notice as written.
- **Path C** — Form 2 with opener variant. The opener references the
  original PR explicitly: "Thanks for the contribution, @{contributor}!
  Adopting your contribution from #{original-pr-number} ...". Posts on the
  NEW PR (the maintainer-created branch). Adoption-notice sentence is
  replaced with `This adopts the contribution from #{original-pr-number}
  on a maintainer-owned branch (the original PR did not have
  maintainer-edits enabled).`
- **Path D** — Form 2 with adoption-notice variant. Posts on the FOLLOW-UP
  PR. Adoption-notice sentence is replaced with `This is a follow-up to
  #{original-pr-number} (already merged), bundling the maintainer fixups
  into a separate PR.` The `{final-review-paragraph}` references the
  follow-up-PR tip, not the original.

## Use from formulas

A formula or skill that wants to render and post this comment should:

1. Gather all inputs listed above (most are mechanical `gh` / `git` /
   filesystem reads; the synthesis and scorecard files come from the
   review run).
2. Choose Form 1 or Form 2 based on whether `maintainer-fixup-log` is empty.
3. Render each `{model-rendered}` field via an LLM call against the raw
   inputs.
4. Run the validation gates against the rendered comment.
5. On pass: write to `<run_dir>/outputs/adoption-review.md`, then post via
   `gh pr comment <number> --body-file <path>`.
6. On fail: write rejected text and STOP per the failure-fallback rule.

The `pr-review` pack's `mol-adopt-pr` formula's `finalize` step uses this
template for all four merge paths via this exact procedure.
